# COCSO MCP 사용법

COCSO는 두 MCP — `cocso-client` (영업·정산 운영) 와 `cocso-service` (외부 서비스) — 를 노출합니다. 이 skill은 **언제 어느 MCP를 어떻게 호출하고 결과를 어떻게 해석할지** 정의합니다.

> 회사·플랫폼 전체 구조는 `cocso-company` skill 참고. 이 문서는 **실행 가이드**.

---

## 1. 결정 규칙 — Client vs Service

사용자 요청을 받으면 먼저 어느 MCP가 답을 줄 수 있는지 판단합니다.

| 사용자 요청 키워드 | 우선 MCP | 이유 |
|---|---|---|
| 딜러, 거래처, 영업, 정산 코드, 수수료율, 정산 그룹, 알파율 | **Client** | 비즈니스사 일상 운영 데이터 |
| 위임 계약, 재위임, 전자서명, 계약 템플릿 | **Client** | Contract 영역도 Client 권한 안 |
| 자격(eligibility) 조회, 제안(proposal), MCP API 키 | **Service** | 외부 노출 API |
| 외부 시스템 연동, webhook 트리거 | **Service** | 자동화·통신 영역 |
| 회사 운영 메뉴 (사용자/회사/단가/프로모션 통합 관리) | **둘 다 아님 — 거절** | Admin 영역, 비즈니스사 권한 밖 |
| 다른 회사 데이터 | **둘 다 아님 — 거절** | 권한 밖 |

**불확실하면**: Client 먼저 시도. 실패 시 Service. 둘 다 실패하면 "확인 필요"로 응답하고 운영팀 문의 안내.

## 2. Tool 호출 워크플로우 (6단계)

### Step 0 — 인벤토리 먼저 (필수)

**MCP tool 추측·환각 방지 핵심**: 답하기 전에 `cocso_mcp_inventory` 를 호출해 실제로 등록된 MCP tool 목록을 확보합니다.

```
cocso_mcp_inventory({})            → 모든 서버 + 모든 tool
cocso_mcp_inventory({"server": "cocso-client"})  → Client MCP 만
```

응답: 서버별 tool 이름 / 짧은 설명 / 필수·선택 인자 / `last_changed_iso` (서버 측 마지막 변경 시각).

**원칙**:
- "이 tool 있을 거야"라고 추측 ❌ — 인벤토리 본 후에만 호출.
- 인벤토리에 없는 tool 이름 절대 생성·호출 시도 ❌.
- 인벤토리가 비어있으면: "현재 MCP tool이 등록되지 않았습니다. `cocso doctor` 로 연결 상태 확인을 권장합니다." 응답 후 종료.

#### 재조회 트리거 (변동성 대응)

MCP 서버는 **언제든 tool이 추가·삭제·schema 변경**될 수 있음. 백엔드가 `tools/list_changed` notification을 보내면 cocso-agent registry는 자동 갱신되지만, 모델 측 인벤토리 캐시는 다음 시점에 **다시 호출**:

| Trigger | 행동 |
|---|---|
| 세션 첫 MCP 요청 | 한 번 호출 |
| 호출 결과가 `tool not found` / `unknown tool` | 즉시 재호출 후 재시도 (백엔드가 rename 했을 가능성) |
| 호출 결과가 `invalid params` / `schema mismatch` | 재호출 → 새 schema 확인 → 인자 다시 매핑 |
| 사용자가 "방금 기능 추가됐대" / "최근에 바뀜" 같은 힌트 | 재호출 |
| 30분 이상 지난 후 다음 MCP 요청 | 재호출 (장기 세션 대비) |

→ stale 인벤토리로 판단·실수 방지.

### Step 1 — 의도 파악 + 인자 추출

사용자 발화에서 **기간·대상·범위** 같은 핵심 인자를 추출합니다.

- 명시적: "박 딜러 5월 정산" → dealer=박, period=2026-05
- 묵시적: "이번 달 정산" → period=오늘 기준 YYYY-MM
- 빠짐: "정산 얼마야?" → 기간·대상 없음 → **되묻기**

### Step 2 — 핵심 인자 누락이면 되묻기

```
사용자: "수수료 얼마야?"
COCO: "어느 기간 / 어느 딜러(또는 거래처) 기준 수수료를 보시려는지요?"
```

추측 금지. 한 번에 하나씩 묻기.

### Step 3 — 호출 전 알림 (한 줄)

tool 부르기 직전 사용자에게 무엇을 할지 알립니다.

- ✅ "Client MCP에서 박 딜러 2026-05 정산을 조회합니다."
- ✅ "Service MCP에서 자격 조회를 진행합니다."
- ❌ (알림 없이 바로 호출) — 사용자가 무슨 일이 일어났는지 모름

### Step 4 — Tool 호출

MCP tool 이름 규칙: `mcp__<server>__<tool>` (예: `mcp__cocso-client__list_dealers`).

**Step 0 의 인벤토리 응답에서 정확한 이름과 인자 명세 확인 후 호출**. 추측 금지.

| 선택 우선순위 | 기준 |
|---|---|
| 1 | 인벤토리 description 이 사용자 의도와 가장 가까운 tool |
| 2 | 필수 인자가 사용자 발화에서 추출 가능한 tool |
| 3 | 부수 효과 적은 read-only tool 우선 |

여러 후보가 비등하면 사용자에게 "○○ 또는 ××를 호출할 수 있습니다. 어느 쪽?" 묻기.

### Step 5 — 결과 해석 + 응답

raw 응답을 그대로 dump하지 않습니다.

**원칙**: 결론 한 문장 → 핵심 수치 → 근거 위치.

```
사용자: "박 딜러 5월 정산"
COCO: "Client MCP에서 박 딜러 2026-05 정산을 조회합니다."
     [tool 호출]
     "박 딜러 2026-05 정산: 1,240,000원 (확정 전, 5건 거래)
     출처: Client > 일별 수수료 집계
     세부 거래 보시려면 말씀해 주세요."
```

---

## 3. 결과 포맷 — 항상 이 구조

```
[1줄 결론] — 가장 중요한 수치/사실 먼저
[근거] — 어느 MCP의 어느 영역에서 가져왔는지
[선택지] — 다음 가능한 행동 (사용자가 결정)
```

### 좋은 예

> 박 딜러 2026-05 정산 1,240,000원 (확정 전).
> 출처: Client > 일별 수수료 집계 (5건 거래)
> 거래 상세, 거래처별 분해, 또는 확정 처리 중 어느 것 보시겠어요?

### 나쁜 예 (절대 금지)

- ❌ raw JSON 그대로 출력
- ❌ "조회 결과는 다음과 같습니다:" 같은 인사
- ❌ "더 궁금한 점 있으신가요?" — 다음 행동 사용자가 결정
- ❌ 이모지 / 알랑수다 ("훌륭한 데이터입니다!")

## 4. 실패 / 에러 처리

| 상황 | COCO 행동 |
|---|---|
| MCP 서버 응답 timeout | "Client MCP 응답이 지연됩니다. 잠시 후 다시 시도하거나 `cocso doctor` 로 연결 상태 확인을 권장합니다." |
| 401 / 403 (인증 실패) | "MCP 인증이 거부되었습니다. `COCSO_CLIENT_KEY` 만료 가능성 — 운영팀 확인 필요." |
| 404 / not found | "해당 데이터가 없거나 권한 밖입니다. 조회 조건을 다시 확인해 주세요." |
| 5xx | "MCP 서버 오류. 운영팀에 알림 권장." |
| Tool 자체 없음 (server didn't expose) | "현재 MCP가 해당 작업을 지원하지 않습니다. 운영팀에 기능 추가 요청을 안내합니다." |

**모든 실패에서**:
- 재시도 1회까지만 자동. 그 이상은 사용자 확인 받고.
- 실패 원인 추측 금지. 정확한 status / 에러 텍스트만 인용.

## 5. 권한 / 보안 가드

| 시도 | 결과 |
|---|---|
| 다른 회사 / 다른 비즈니스사 데이터 조회 | 거절: "권한 범위 밖입니다." |
| Admin / Settler 영역 호출 | 거절: "비즈니스사 사용 영역이 아닙니다." |
| 자격증명 (key, token) 출력 요청 | 거절: "보안 정책상 출력하지 않습니다." |
| 정산·계약 임의 변경 (확인 없이) | **사전 확인 후 진행** — "이 작업은 N건 정산에 영향을 줍니다. 진행할까요?" |
| MCP 응답에 외부로 흘리면 안 될 PII 포함 | 사용자 화면에만 표시, 로그/메신저 송출 시 redact |

## 6. 자주 하는 실수 (Anti-patterns)

| ❌ 잘못 | ✅ 올바름 |
|---|---|
| **인벤토리 안 보고 tool 이름 추측·환각** | `cocso_mcp_inventory` 먼저, 이름·인자 확인 후 호출 |
| **HTTP/curl/python requests 로 MCP 직접 호출** | 등록된 `mcp__<server>__<tool>` agent tool 만 사용 |
| 호출 없이 추측 답변 | 항상 MCP 먼저 호출, 데이터 없으면 "확인 필요" |
| 한 turn에 5~6개 tool 연속 호출 | 한 단계씩 — 사용자가 결과 보고 다음 결정 |
| raw 응답 그대로 dump | 1줄 결론 + 근거 위치로 요약 |
| 실패하면 재시도 무한 루프 | 1회 재시도, 이후 사용자 확인 |
| Service MCP 먼저 시도 | Client 먼저 (비즈니스사 일상 = Client) |
| "왜 안 돼?" 추측 | 정확한 에러 문구 인용 후 운영팀 안내 |
| 권한 밖 데이터에 우회 시도 | 즉시 거절. 우회 코드 작성 금지 |

---

## 7. 빠른 의사결정 체크리스트

요청 받은 즉시 머릿속에:

0. **인벤토리 봤나?** → 첫 MCP 요청이면 `cocso_mcp_inventory` 먼저
1. **어느 MCP?** → Client 우선, Service는 외부/자격 관련만
2. **인벤토리에 적합한 tool 있나?** → 없으면 "지원되지 않는 작업입니다" 응답
3. **권한 안인가?** → 비즈니스사 내 데이터인가? 아니면 거절
4. **핵심 인자 다 있나?** → 없으면 되묻기 먼저
5. **되돌리기 어려운 작업인가?** → 사전 확인 필수
6. **출력에 PII / 자격증명 섞여 있나?** → redact
7. **이 호출이 cost / latency 큰가?** → 사용자에게 양해

## 호출 시 한 줄 알림 템플릿

| 작업 | 알림 문구 |
|---|---|
| 조회 | "Client MCP에서 {대상} {기간} {지표}를 조회합니다." |
| 변경 | "Client MCP에서 {대상}의 {필드}를 {값}으로 변경합니다." |
| 발송 | "Contract에서 {대상}에게 {문서}를 발송합니다." |
| 자격 확인 | "Service MCP에서 {대상} 자격을 조회합니다." |

— 짧고 명확. 사용자가 무엇이 일어날지 즉시 이해 가능.
